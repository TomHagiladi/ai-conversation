## 5. אבטחת GitHub

**משתמש:** מה לגבי גיטהאב? איך אני שומר שדברים שהעלתי לשם לא יחשפו אותי לפרצה

**Claude:** שאלה מצוינת. GitHub הוא מקום שבו הרבה אנשים נשרפו - בוא ניכנס לפרטים.

### 1. הסיכון הכי נפוץ: סודות בקוד

זה הקטסטרופה הקלאסית - מפתחות API, סיסמאות, טוקנים שנדחפו בטעות.

**מה קורה גם אם מחקת:**
- `git rm secret.env && git commit` **לא מוחק** את הסוד. הוא נשאר בהיסטוריה
- בריפו פאבליק - בוטים סורקים את GitHub תוך **שניות** מהדחיפה. אם דחפת מפתח AWS, יש סבירות גבוהה שמישהו כבר אסף אותו
- אפילו ריפו שהיה פאבליק לרגע ואז עשית privacy - יכול להיות שכבר נסרק

**מה לעשות אם זה קורה:**
1. **תרוקן את הסוד מיד** - ב-AWS/Anthropic/Stripe וכו', לבטל את המפתח ולהנפיק חדש. זה הצעד הקריטי
2. רק אז - לנקות מהיסטוריה (`git filter-repo` או BFG Repo-Cleaner) ולעשות force push
3. הנחת היסוד: **"if it touched git, it's compromised"**

### 2. הגנות מנע

**לפני שאתה דוחף**
- **`.gitignore` רציני**: `.env`, `*.pem`, `*.key`, `credentials*`, `.aws/`, `secrets/`, `id_rsa*`
- **pre-commit hooks**: `gitleaks`, `trufflehog`, `detect-secrets`. רצים לפני כל commit וחוסמים סודות
- **תבנית `.env.example`** במקום `.env` - שיהיה ברור איזה משתנים נדרשים בלי לחשוף את הערכים

**ב-GitHub עצמו**
- **Push Protection** (חינם, גם לריפוז פרטיים): GitHub חוסם push אם הוא מזהה סוד מוכר. **תפעיל את זה בכל ריפו**. Settings → Code security → Secret scanning → Push protection
- **Secret Scanning Alerts**: סורק ריפוז קיימים על סודות. אם משהו מצא - אתה מקבל התראה
- **Dependabot**: סורק תלויות פגיעות (CVEs בספריות שאתה משתמש בהן)
- **CodeQL / Code Scanning**: סטטית אנליזה לחולשות בקוד עצמו (SQL injection, XSS וכו')

### 3. הרשאות וגישה

**Personal Access Tokens (PATs)**
- **Fine-grained tokens** > Classic tokens. תמיד. fine-grained מאפשרים להגביל לריפו ספציפי, scope מינימלי, expiry
- **תפוגה קצרה**: 30-90 ימים מקסימום. אל תיצור tokens "no expiration"
- **scope minimal**: אם הכלי צריך רק לקרוא issues, אל תיתן `repo` שלם
- **rotation**: לסובב באופן קבוע, ובוודאי אם משהו דלף

**SSH Keys**
- **passphrase על המפתח** - אם הלפטופ נגנב, המפתח לא יעבוד מיד
- **מפתח נפרד לכל מכשיר** - לא להעתיק את אותו `id_ed25519` בין מחשבים
- **ed25519 > RSA** למפתחות חדשים

**OAuth Apps & GitHub Apps**
- כל פעם שאתה מאשר "Authorize App" באיזשהו שירות - אתה נותן לו גישה לחשבון שלך
- **לבדוק תקופתית**: Settings → Applications → ראה מי מורשה. למחוק כל מה שלא משתמש
- ה-MCP servers שאתה רואה בסשן הזה נכנסים לקטגוריה הזו

**Branch Protection**
על הענפים החשובים (`main`, `production`):
- Require pull request reviews
- Require status checks
- Block force push
- Require signed commits (אם אתה ממש פרנואיד)

### 4. 2FA וחשבון

- **תמיד 2FA** - GitHub כבר אוכף את זה לרוב המשתמשים. **passkey** או **hardware key** (YubiKey) > SMS > TOTP app > SMS
- **Recovery codes** - לשמור במקום בטוח (1Password וכו'), לא בקובץ טקסט בדסקטופ
- **Sessions tab** - לראות מאיפה אתה מחובר. לחתוך כל מה שלא מזוהה

### 5. ריפוז פרטיים ≠ פרטיים לחלוטין

- **כל מי שעבד אצלך** עדיין יש לו clone מקומי. גם אם מחקת, אצלו זה קיים
- **GitHub Actions logs** - יכולים לחשוף ערכים אם עשית `echo $SECRET` בטעות. תשתמש ב-`::add-mask::` ובסודות אמיתיים של Actions
- **Forks**: בריפו פאבליק, אם מישהו פירק והדחפת סודות - הסוד נשאר ב-fork גם אחרי שמחקת אצלך
- **Gists** - גיסטים "secret" הם לא פרטיים, פשוט לא מאונדקסים. URL = גישה

### 6. CI/CD

GitHub Actions זה משטח התקפה משמעותי:

- **Secrets ב-Actions** (Settings → Secrets) - לאחסן שם, לא בקוד. הם מוסתרים מהלוגים אוטומטית
- **`pull_request_target`** - מסוכן, נותן לקוד מ-PR גישה לסודות. תבחר `pull_request` רגיל
- **Third-party actions** - `uses: someone/action@main` יכול להחליף לוגיקה מתחתיך. **תקבע SHA**: `uses: someone/action@a1b2c3d...`
- **OIDC במקום סודות סטטיים** - לעבודה מול AWS/GCP/Azure, OIDC טוב יותר ממפתח שמור

## 7. בקשת המוצר: סוכן שומר

**משתמש:** אני רוצה ליצור סוכן שכל הזמן שומר עלי בכל פרויקט והוא בשקט ברקע תמיד הכל שיחה שלי עם קלוד קוד בכל מקום בה אני מקיים אותה ואם הוא מזהה מידע רגיש הוא יוצא משתיקתו ומייעץ לי, מזהיר אותי, עוצר אותי מלעשות טעות שאינני מודע אליה

**Claude:** רעיון מצוין ובר-ביצוע. הכלי המתאים בדיוק קיים ב-Claude Code - **Hooks**.

### הכלי הנכון: Hooks

Claude Code מאפשר להגדיר ב-`~/.claude/settings.json` סקריפטים שרצים אוטומטית בנקודות-חיים מסוימות בכל סשן, בכל פרויקט. ההוקים האלה הם **לא** סוכן AI - הם פקודות shell שאתה שולט בהן. זה בדיוק ה-"שומר השקט".

### נקודות הציד הרלוונטיות

| Hook | מתי רץ | יכולת חסימה |
|------|--------|-------------|
| `UserPromptSubmit` | לפני שההודעה שלך נשלחת לאנטרופיק | כן - יכול לבטל את ההודעה |
| `PreToolUse` | לפני שאני מריץ Read/Bash/Write/וכו' | כן - יכול לחסום את הפעולה |
| `PostToolUse` | אחרי שכלי רץ | לא, אבל יכול להזהיר |
| `SessionStart` | בתחילת סשן | טעינת קונטקסט |
| `Stop` | בסוף תגובה | בדיקה סופית |

**הנקודה הקריטית**: `UserPromptSubmit` רץ **לפני** שההודעה עוזבת את המחשב שלך. אם הוא חוסם - אנטרופיק לא רואים את התוכן בכלל.

### מה השומר יזהה

**שכבה 1: Regex דטרמיניסטי (מהיר, אפס הזיות)**
- AWS keys: `AKIA[0-9A-Z]{16}`
- GitHub tokens: `gh[pousr]_[A-Za-z0-9]{36,}`
- OpenAI/Anthropic keys: `sk-[A-Za-z0-9]{20,}`, `sk-ant-...`
- Stripe: `sk_live_`, `pk_live_`
- Private keys: `-----BEGIN (RSA|OPENSSH|EC) PRIVATE KEY-----`
- JWT, Slack tokens, Google API keys, וכו'
- שורות `.env`-style: `PASSWORD=...`, `SECRET=...`, `TOKEN=...`

**שכבה 2: Entropy detection** - מחרוזות אקראיות ארוכות

**שכבה 3: בדיקות מבוססות-נתיב** - חסימה של `.env*`, `*.pem`, `id_rsa*`, `~/.aws/credentials`, `~/.ssh/`, `credentials.json`, וכו'

**שכבה 4 (אופציונלית): סנטיקה דרך LLM** - שולח קטעים חשודים ל-Claude Haiku לקבלת חוות דעת

### הזרימה האמיתית

```
אתה כותב prompt
   ↓
UserPromptSubmit hook (סורק את ההודעה)
   ↓
   ├─ נקי? → ההודעה נשלחת אליי
   └─ חשד? → "⚠️ זיהיתי AWS key. לאשר/לבטל/לערוך?"
   ↓
אני רוצה לקרוא קובץ
   ↓
PreToolUse hook (סורק את הנתיב/הפקודה)
   ↓
   ├─ נקי? → אני קורא
   └─ ".env"? → חוסם, אומר לי "הקובץ הזה מכיל סודות"
   ↓
הכלי רץ, מחזיר תוכן
```

### איך זה עובד "בכל מקום"

- **בלוקאלי**: ההגדרה ב-`~/.claude/settings.json` חלה על **כל** הפרויקטים שאתה פותח על המחשב הזה
- **בענן**: ה-settings.json לא פרסיסטנטי בין סשנים. אפשרות: SessionStart hook שמתקין את השומר אוטומטית בכל סשן חדש
- **לוגים מרכזיים**: השומר יכול לכתוב audit trail (אופציונלי)

### מימוש מומלץ

מבנה קבצים:
```
~/.claude/
├── settings.json              ← הגדרת ה-hooks
└── hooks/
    └── secret-guard.py        ← הסקנר הראשי
```

ממליץ **Python** ולא bash - Regex מורכבים, יותר קל לבדוק, יותר קל להוסיף שכבת LLM בעתיד.

### מגבלות שחשוב להכיר

1. **לא תופס הכל**: סוד "מנופף" בעין (סיסמת `Apollo123!`) לא יזוהה ע"י entropy או regex. רק LLM יזהה את זה
2. **False positives**: יקרו. תצטרך allowlist mechanism
3. **ביצועים**: אם הסקנר איטי - הוא יעצבן. צריך לשמור על <100ms
4. **אתה עדיין יכול לדרוס**: אם השומר חוסם ואתה אומר "כן בטוח, שלח" - הוא ישלח. הוא מזהיר, לא בית סוהר
5. **רק על המחשב הזה**: שומר על הלפטופ לא רואה מה אתה עושה מהנייד
6. **לא מגן מפני אנטרופיק עצמם**: הוק רץ לפני שהמידע נשלח, אבל ברגע שהמידע נשלח (כי אישרת), הוא אצלם
7. **prompt injection**: PreToolUse יכול לחסום פקודות מסוכנות (rm -rf, curl | sh) - זו הגנה משמעותית

### ההצעה: 3 שלבים

**שלב 1 (MVP, שעה)**: Python script + `UserPromptSubmit` + `PreToolUse` hooks. רגקסים בסיסיים + חסימה של נתיבים רגישים
**שלב 2 (אופציונלי)**: PostToolUse + audit log + allowlist
**שלב 3 (אופציונלי)**: שכבת LLM סמנטית, סנכרון בין מכונות דרך GitHub

### שאלות לפני התחלה

1. **איפה תתקין?** הסשן הזה בענן - לא יעזור לך. לדחוף ל-repo ולמשוך ללפטופ?
2. **רמת קפדנות**: **חסימה קשה** (דורש אישור מפורש להמשיך) או **אזהרה רכה**?
3. **לוגים**: לרשום audit log של כל אזהרה, או רק להציג ולשכוח?
4. **שכבת LLM עכשיו או אחר כך?**

## נספח: עקרונות מנחים שעלו בשיחה

1. **GitHub הוא נקודת המפגש** - בין סשן ענן ללפטופ. בלעדיו - אין גשר
2. **כל מה שעובר דרך AI עובר דרך הספק** - גם בלוקאלי. הקוד נקרא לוקאלית, אבל התוכן נשלח ל-API
3. **MCP servers הם דלת רחבה** - שווה תשומת לב מיוחדת. OAuth tokens, prompt injection
4. **Push Protection + Secret Scanning של GitHub** - חינם, חובה
5. **Fine-grained PATs > Classic** - תמיד
6. **"if it touched git, it's compromised"** - הנחת מוצא נכונה
7. **לא ניתן לאמת מלאה AI** - אבל ניתן לבנות ארכיטקטורת אמון: least privilege, code review, audit logs, rotation
8. **Hooks = שומר אמיתי** - לא AI חכם, סקריפט דטרמיניסטי שאתה שולט בו, רץ אוטומטית בכל סשן
9. **UserPromptSubmit חוסם לפני השליחה** - זה ההבדל הקריטי בין "מזהה" ל"מונע"
10. **קולגה ג'וניור מוכשר** - מטאפורה נכונה לעבודה עם AI: עוזר חזק, לסקור לפני merge, בלי גישת פרודקשן
